Three small refinements that will stick.

After looking at the current prototype in its tabbed state, I noticed three places where the design is pulling its punches — tab headers mixing type systems, the "Where the money goes" section still carrying a generic stacked bar, and the charts inheriting rectangular shapes that fight the warm, paper-first palette. Here's what I'd change, and how each maps onto your React codebase.

01 · Font consistency 02 · Where the money goes 03 · Lines & bars

01 · Font consistency at the top of every tab.

The tabs today mix three treatments — some use Instrument Serif at 22/24/56px, some use Inter at 32/font-semibold, some pair a "caps eyebrow" and some skip it. The fix is one rule, applied everywhere.

The rule: every page/tab has exactly one H1 — Instrument Serif 48 / letter-spacing −2% / line-height 1.0 — optionally preceded by a caps eyebrow (Inter 500, 10px, +14% tracking, ink-3). Subsection headings inside a page step down to serif 22. Inter is never used for headings.

Implementation

  1. Create <PageHeader eyebrow title subtitle hero? /> in shared/PageHeader.tsx. All views render this component instead of their own H1 — eliminates the three current patterns.
  2. Lock the type scale in CSS variables: --h1: 48px, --h2: 22px, --hero: 56px, --caps: 10px. Remove every inline text-[NNpx] on heading elements and replace with the token classes.
  3. Delete all font-semibold on headings. Instrument Serif at regular weight is heavier than Inter-600 optically already; pairing them looks muddy. Keep font-semibold reserved for tabular values only.
  4. One accent move per page. Either an italic word ("across") or a teal-colored number — never both.
02 · Where the money goes

Not a bar — a ledger with proportions.

The current stacked bar is a spectrum of teals packed into 12 pixels of height. You can't read it. The fix is to lean into what this product is — a ledger — and show the data as a sorted list with integrated proportion bars. A small donut anchors the total in the corner.

Why it works: the donut gives a single-number headline ("62% goes to the mortgage"), while the ledger rows let the user read the actual P&L line by line — sorted largest to smallest, with the thin proportion rule doing visual work that's secondary to the numbers. It reads as an accountant's statement, not an infographic.

Implementation (ExpenseBreakdown.tsx)

  1. Replace the 12px stacked bar with an SVG donut. Use a single radius, 10px stroke, rotate −90°. Each segment is a <circle> with stroke-dasharray — no library needed.
  2. Sort rows descending by value before rendering; mortgage will always anchor first.
  3. The 2px proportion rule under each label is bounded by the widest row (mortgage = 100%), not by container width. Every other row reads as a fraction of the biggest, which is the intuitive question ("how small is this vs. the mortgage?").
  4. Donut center text should be generated: find the largest segment, show "XX% goes to [label]." Single source of truth.
03 · Line & bar charts

Thinner strokes, less ink, more paper.

Your current charts feel like Chart.js defaults — thick strokes, solid area fills, axis labels in the body font. The system wants the opposite: hair-thin strokes, translucent wash fills, mono labels for numbers, and every line anchored to a meaningful end value.

The chart system — applied consistently

  1. Strokes. Line charts use stroke-width: 1.25 for secondary series, 1.5 for the hero series. No chart line should be thicker than 2px. Dashed only for projections or "budget" lines.
  2. Fills. Area charts use a linear gradient from stop-opacity: 0.15 → 0. Never a solid fill under a line.
  3. Labels. Every axis label is JetBrains Mono 10px ink-3. Every annotation (callouts, event labels) is Instrument Serif italic 10px ink-2. This split makes the type do the semantic work: mono = data, serif = commentary.
  4. End-value markers. Every line terminates in a filled dot + right-of-dot label showing the final value and series name. Kills the need for a separate legend in most cases, and turns the chart into a self-contained summary.
  5. Bars. Horizontal only, with a centered "base case" rule. Cashflow scenarios render as offsets from base; color carries direction (teal = good, danger red = bad); the base row gets a warm wash. Removes the need for a Y axis.
  6. Seasonal bars (SeasonalBars.tsx) keep the rectangular form but drop to width: 100% of the slot (no flex-gap), use 3 semantic colors (ink = peak, accent = shoulder, rule-strong = off), and render values as tiny mono labels above each bar — same as the current sparkline callouts.
  7. No Chart.js, no Recharts. All of this is < 80 lines of SVG per component. A shared <ChartFrame> wrapper provides the gridlines + axis labels; each chart type is a small presentational component on top.
Shared component to build first: shared/Chart.tsx exporting <LineChart series={[…]} xLabels={[…]} />, <SensitivityBars rows={[…]} base={…} />, and <SeasonalBars months={[…]} />. Everything in the app — Results tab projections, Sensitivity tab, Revenue tab seasonality — consumes those three. Current components in shared/SparklineAmort.tsx, SeasonalBars.tsx, SensitivityCard.tsx get deleted and replaced.
